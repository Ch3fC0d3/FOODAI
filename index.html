About the Project

This thesis proposes an inverse design approach to computational gastronomy, developing an AI system that generates novel recipes based on targeted functional and sensory outcomes (e.g., texture, dietary constraints). Using computational gastronomy, pharmaceutical modeling and taste prediction algorithms, to reverse the traditional forward modeling of recipes, the system maps desired properties to compatible ingredient and preparation strategies, offering a new framework for data-driven, function-first food innovation.

üß™Methodology

Define Functional Targets

Create a structured schema for "desired outcomes" (e.g., fluffy, low-GI, crispy vegan). Use food science texts to associate these with known functional variables (moisture retention, binding, leavening, etc.).

Ingredient & Technique Database

Build a dataset (or repurpose RecipeDB, FlavorDB) tagged by:

  • Ingredient functional roles (binder, emulsifier, leavener)
  • Texture outcomes (crispy, chewy, airy)
  • Dietary flags (gluten-free, vegan, paleo)
  • Preparation methods (baking, steaming, frying)

Design the AI System

Two key options:

Constraint-based recommender system

Match functional targets to ingredient roles using a knowledge graph or rule-based logic.

Neural network with reverse-mapping

Use a generative model (e.g., GPT-2/Neo fine-tuned on recipe data) + prompt engineering to "solve backward" from the functional goal.

Prototype & Validate

Generate recipes for 3‚Äì5 specific, non-trivial functional targets (e.g., "crispy, eggless, high-fiber pastry").

Conduct blind sensory tests or get chef feedback kitchen outcomes.

Include a comparison to baseline (e.g., "existing gluten-free recipes" vs. "AI-generated").

Evaluation Metrics

  • Technical validity (ingredient functionality checks)
  • Novelty (ingredient combinations that don't exist in known datasets)
  • Sensory feedback (human perception data, if possible)
  • Computational performance (runtime, solution diversity)

2. System Architecture

2.1 Overview

The system follows a modular design comprising:

  • User Interface Layer ‚Äì Receives user-defined sensory and functional targets.
  • Application Programming Interface (API) ‚Äì Mediates data exchange between frontend and backend.
  • AI Engine ‚Äì Executes inverse mapping from sensory goals to ingredients and preparation steps.
  • Databases ‚Äì Store ingredients, recipes, nutrition facts, and sensory mappings.
  • External Integrations ‚Äì Interfaces with third-party APIs for recipe/nutritional data.

2.2 Data Flow Summary

  1. User Input: Functional goals (e.g., texture, flavor, dietary needs)
  2. Request Handling: API routes inputs to processing engine
  3. Computation: AI model selects ingredients + preparation logic
  4. Data Retrieval: Queries internal and external databases
  5. Output: Returns recipe with rationale and nutritional breakdown

2.3 Third-Party Integrations

  • FlavorDB ‚Äì Sensory compound mapping
  • Edamam API ‚Äì Nutritional and dietary validation
  • Firebase/Auth0 ‚Äì User authentication and secure login
  • Figma or Adobe XD ‚Äì UI/UX prototyping and design
